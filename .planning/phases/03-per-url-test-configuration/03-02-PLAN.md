<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CheckConfig → ValidationService | Check type strings from config used as filter keys against ValidationResult.type |
| ValidationService → ScraperEngine | Filtered validation results flow back to caller; fewer results = fewer alerts |
| scraper.py → filesystem | CheckConfig.load() reads config/url_checks.yaml from disk at run() time |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-05 | Tampering | check_config.enabled_checks_for() used as filter in validate_course() | accept | Filter uses frozenset membership test (O(1), no code execution). The check type strings from config are compared against ValidationResult.type strings from validators — neither path executes arbitrary code. |
| T-03-06 | Denial of Service | check_config=None bypass in validate_course() | accept | None means "run all checks" — this is an intentional backward-compat default, not a security bypass. ValidationService is not an internet-facing service; it runs in a trusted local process. |
| T-03-07 | Elevation of Privilege | base_url from SQLite DB used as config key | accept | DB is a local SQLite file written only by the scraper process itself. base_url values come from urls.txt which is operator-controlled. Not internet-user-supplied input. |
| T-03-08 | Tampering | check_config loaded once in run() and reused for recheck pass | accept | CheckConfig is a Pydantic model (immutable in practice). Re-using the same object across both validation passes is intentional and correct — the config should not change mid-run. |
</threat_model>
